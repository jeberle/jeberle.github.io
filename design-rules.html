Design Rules

  • It’s already been written, tested, and documented. The trick is to find and incorporate it.
  • Code is evil, minimize it
    • Copy/Paste is evil, find another way
    • Duplicate code is worse than redundant data
    • No commented-out code blocks
    • No dead code
    • No single-use string constants
    • Avoid wrapping if possible
    • Avoid layering if possible
  • State is evil, minimize it
    • Minimize global variable use
    • Minimize member variable use
    • Minimize variable visibility
  • Keep code and data separate
    • Functions work
    • Structs work
    • Classes work when functions or structs cannot be used
  • Use the right tool for the job
    • Know your options
    • Match the language to the problem
  • Fight the good fight
    • “Later” never comes, as in the phrase, “I’ll fix it later”
    • Expand tabs
    • Split long lines
  • Meet the configuration at the door, don’t invite him in
    • Any time you have to configure your configuration, you have a problem on your hands
    • XML bad, INI good
